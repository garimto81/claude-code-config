# PRD 초보자 가이드
*처음 PRD를 작성하는 주니어 개발자를 위한 단계별 가이드*

**버전**: 1.0.0 | **타겟**: 초급 개발자 | **소요**: ~40-60분

---

## 🎓 PRD란 무엇인가요?

**PRD (Product Requirements Document)**는 개발할 기능이 **무엇**인지, **왜** 필요한지, **어떻게** 작동해야 하는지 설명하는 문서입니다.

### 왜 PRD가 필요한가요?
- ✅ 개발 전 요구사항 명확화
- ✅ 불필요한 재작업 방지
- ✅ 팀원 간 이해 공유
- ✅ 테스트 기준 마련

---

## 📝 작성 프로세스

### Step 1: 요구사항 수신
```
예시: "사용자가 프로필을 편집할 수 있게 해주세요"
```

### Step 2: 명확화 질문 (옵션 선택)
아래 질문에 **A/B/C/D**로 답변하면 됩니다!

### Step 3: PRD 템플릿 작성
답변을 바탕으로 템플릿을 채웁니다.

### Step 4: 검토 & 승인
작성한 PRD를 확인하고 개발을 시작합니다.

## 📄 PRD 템플릿 (단계별 작성)

### Section 1: 기본 정보 (5분)

```markdown
# [기능 이름] PRD

**번호**: PRD-0001
**작성일**: 2025-01-12
**작성자**: [이름]
**우선순위**: Medium
**상태**: Draft
```

**채우는 방법**:
- 번호: `PRD-0001`, `PRD-0002`, ... (순서대로)
- 작성일: 오늘 날짜
- 우선순위: High (급함) / Medium (보통) / Low (여유)
- 상태: Draft (작성 중) → Approved (승인) → In Progress (개발 중) → Completed (완료)

---

### Section 2: 개요 (10분)

```markdown
## 1. 개요

### 문제 정의
[Q1 답변 기반]
현재 사용자는 [무엇을 못하고 있나요?]
이로 인해 [어떤 불편함이 있나요?]

예시:
> 현재 사용자는 가입 후 프로필 정보를 수정할 수 없습니다.
> 이로 인해 이름이나 이메일이 바뀌어도 업데이트할 방법이 없어 불편합니다.

### 제안 솔루션
[어떻게 해결할 건가요?]

예시:
> 프로필 편집 기능을 추가하여 사용자가 이름, 이메일, 전화번호를 직접 수정할 수 있게 합니다.

### 예상 효과
[Q7, Q1 답변 기반]

예시:
- 사용자 만족도 20% 향상
- 고객 지원 문의 30% 감소
```

**채우는 방법**:
1. **문제 정의**: 현재 상황 + 불편함
2. **제안 솔루션**: 해결 방법 (1-2문장)
3. **예상 효과**: 측정 가능한 숫자로

### Section 5: 사용자 스토리 (15분)

```markdown
## 4. 사용자 스토리

### 스토리 1: [Q6의 시나리오 1]
As a [사용자 유형],
I want to [수행할 작업]
So that [얻을 이익]

**수락 기준** (Acceptance Criteria):
- Given [조건]
- When [행동]
- Then [결과]

예시:
### 스토리 1: 이름 수정
As a 일반 사용자,
I want to 프로필 페이지에서 이름을 수정
So that 결혼 후 바뀐 성을 반영할 수 있다

**수락 기준**:
- Given 사용자가 로그인되어 있고
- When 프로필 페이지에서 이름을 변경하고 저장 버튼을 클릭하면
- Then 새 이름이 데이터베이스에 저장되고 화면에 즉시 반영된다

### 스토리 2: [Q6의 시나리오 2]
[반복]

### 스토리 3: [Q6의 시나리오 3]
[반복]
```

**작성 팁**:
- **Given**: 시작 조건 (예: 로그인됨, 특정 페이지에 있음)
- **When**: 사용자가 하는 행동
- **Then**: 예상 결과

### Section 6: 기능 요구사항 (15분)

```markdown
## 5. 기능 요구사항

### 5.1 필수 기능 (Must Have)
[Q4, Q6 기반으로 반드시 있어야 하는 기능]

1. **이름 편집**
   - 설명: 사용자가 자신의 이름을 수정할 수 있다
   - 수락 기준:
     - [ ] 이름 필드가 수정 가능하다
     - [ ] 빈 값으로 저장 불가 (최소 2자)
     - [ ] 저장 성공 시 메시지 표시

2. **이메일 편집**
   - 설명: 사용자가 이메일 주소를 변경할 수 있다
   - 수락 기준:
     - [ ] 유효한 이메일 형식 검증
     - [ ] 중복 이메일 확인
     - [ ] 이메일 인증 링크 발송

3. **변경사항 저장**
   - 설명: 수정한 정보를 데이터베이스에 저장
   - 수락 기준:
     - [ ] 저장 버튼 클릭 시 API 호출
     - [ ] 성공 시 "저장되었습니다" 메시지
     - [ ] 실패 시 에러 메시지 표시

### 5.2 선택 기능 (Nice to Have)
[있으면 좋지만 없어도 되는 기능]

1. 변경 이력 조회 (언제 무엇을 바꿨는지)
2. 프로필 사진 업로드 (나중에 추가)
```

**작성 팁**:
- **필수 기능**: 이게 없으면 기능이 작동 안 함
- **선택 기능**: 있으면 좋지만 v1.0에서는 생략 가능

## 🆘 자주 묻는 질문

### Q: PRD를 얼마나 자세히 써야 하나요?
**A**: 주니어 개발자가 읽고 구현할 수 있을 정도면 충분합니다. 코드 레벨까지 상세히 쓸 필요는 없어요.

### Q: 기술 용어를 모르겠어요
**A**: 모르는 용어는 일단 "기존 프로젝트 패턴 따름"으로 작성하고, 시니어에게 검토 요청하세요.

### Q: 수락 기준을 어떻게 쓰나요?
**A**: "테스트할 수 있는 조건"으로 작성하세요.
- ❌ "이쁘게 만든다"
- ✅ "저장 버튼 클릭 시 '저장되었습니다' 메시지 표시"

### Q: 범위가 너무 큰 것 같아요
**A**: Phase로 나누세요!
- Phase 1 (MVP): 최소 기능
- Phase 2: 추가 기능
- Phase 3: 고급 기능
